Ecco il piano per risolvere il problema della perdita di contesto.

### Analisi del Problema
Ho individuato la causa esatta nel file `src/hooks/useAppTranslation.ts`.
Attualmente, la funzione `checkDependency` contiene questa logica:

```typescript
const sequentialContext = aiSettings.sequentialContext ?? true;
const concurrency = Number(aiSettings.translationConcurrency) || 1;

// BUG: Se la concorrenza è > 1, il sistema IGNORA sequentialContext
if (concurrency > 1 || !sequentialContext) {
    return true; // Sblocca immediatamente la pagina successiva
}
```

Questo significa che se hai impostato `Concurrency` (traduzioni parallele) a un valore superiore a 1 (es. 2 o 5), il sistema **disabilita silenziosamente** il controllo sequenziale per favorire la velocità. Di conseguenza, la Pagina 14 viene avviata mentre la 13 è ancora in elaborazione, trovando quindi il contesto "vuoto" (0 caratteri).

### Soluzione Proposta
Modificherò la logica per far sì che l'impostazione `sequentialContext: true` venga **sempre rispettata**, indipendentemente dal livello di concorrenza.

**File:** `src/hooks/useAppTranslation.ts`

**Modifica:**
Rimuovere il controllo `concurrency > 1` dalla condizione di bypass.

```typescript
// Vecchio codice (Bug)
if (concurrency > 1 || !sequentialContext) { return true; }

// Nuovo codice (Fix)
// Se l'utente vuole il contesto sequenziale, dobbiamo aspettare la pagina precedente,
// anche se questo rallenta l'esecuzione parallela.
if (!sequentialContext) { return true; }
```

### Effetti
1.  **Coerenza Garantita**: La Pagina 14 attenderà *sempre* il completamento della Pagina 13 prima di partire.
2.  **Context**: Il log `previousContextChars` tornerà a mostrare il valore corretto (es. ~7000 caratteri) invece di 0.
3.  **Performance**: Se `sequentialContext` è attivo, la traduzione avverrà effettivamente una pagina alla volta (perché ogni pagina dipende dalla precedente), anche se la concorrenza è impostata alta. Questo è il comportamento corretto per garantire la qualità letteraria.
