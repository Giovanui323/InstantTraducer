Based on the analysis of the logs and the codebase, I have identified the following issues and their solutions:

### 1. "STALLED" Log Spam & False Positives
The logs show repeated `[WARNING] STALLED: All X pages blocked` messages. This happens because:
- The "Translate All" function adds pages one by one, triggering `pumpQueue` for *every* page.
- `pumpQueue` logs a warning whenever the head of the queue is blocked by dependencies (e.g., Page 10 waiting for Page 9), even if the system is working correctly (Page 9 is active).
- This creates noise and falsely indicates a problem when the system is just respecting sequential order.

### 2. Inefficient Batch Enqueueing
The current "Translate All" implementation in `App.tsx` iterates through all pages and calls `enqueueTranslation` for each one. This triggers the queue processing logic ($N$) times, causing unnecessary overhead and the log spam mentioned above.

### 3. Queue Stall Logic Refinement
The queue considers itself "stalled" if the first page cannot proceed. However, if other pages are currently active (processing/verifying), the system is **not** stalled; it is simply busy. The warning should only appear if the queue is blocked AND no pages are active (true deadlock).

## Implementation Plan

### 1. Optimize `useTranslationQueue.ts`
- **Add `enqueueMultipleTranslations`**: Create a new function to accept an array of pages. This allows adding multiple pages to the queue while triggering `pumpQueue` only **once** at the end.
- **Refine Stalled Logging**: Update the `pumpQueue` logic to only log a `WARNING` if `activePages === 0`. If pages are active, log a `DEBUG` or `INFO` message indicating the queue is waiting for dependencies to clear.

### 2. Update `useAppTranslation.ts`
- Expose the new `enqueueMultipleTranslations` function from the inner hook to the application layer.

### 3. Update `App.tsx`
- Modify the "Traduci tutto" (Batch Processing) logic to use `enqueueMultipleTranslations` instead of the loop `enqueueTranslation`. This will instantly eliminate the log flood and improve UI responsiveness during batch operations.

This plan directly addresses the "reported issues" visible in the logs (Spam, Stalled warnings) and improves the overall robustness of the queue management.
