I will modify `src/App.tsx` to fix the redundant and cyclic rendering issue.

The problem is caused by a feedback loop between the background page dimension scanning (`scanDims`) and the `ReaderView` component, combined with a small image cache size.
1. `scanDims` updates page dimensions, causing `App` to re-render.
2. `ReaderView` reacts to dimension changes, potentially shifting the layout and triggering `onActivePageChange` with a different page number (e.g., flickering from 8 to 7).
3. This change in `currentPage` triggers `ensureImageLoaded` for the new page.
4. If the image cache is full (currently only 12 images), the previous page's image might be evicted.
5. When the layout stabilizes or the user scrolls back, the process repeats, causing cyclic re-rendering.

To fix this, I will:
1.  **Increase Image Cache Size**: Increase `MAX_IMAGE_CACHE_SIZE` from 12 to 50 to prevent aggressive eviction of images during rapid page changes or layout shifts.
2.  **Add Tolerance to Dimension Updates**: Modify the `scanDims` logic to only update `pageDims` if the change is significant (e.g., > 0.5px). This prevents trivial updates from triggering re-renders.
3.  **Split Dimension Scanning**: Separate the `scanDims` effect into two parts:
    *   **Priority Scan**: Scans the current and adjacent pages immediately when `currentPage` changes.
    *   **Background Scan**: Scans the rest of the document independently of `currentPage` changes, so it doesn't get interrupted and restarted constantly while the user reads.

**Implementation Details:**
-   **File**: `src/App.tsx`
-   **Changes**:
    -   Update `MAX_IMAGE_CACHE_SIZE` constant.
    -   Refactor the `scanDims` `useEffect` into two separate `useEffect` hooks.
    -   Add `Math.abs(prev - new) > 0.5` check in `setPageDims` state updates.

This will stabilize the rendering process, stop the "erratic" behavior, and reduce CPU/memory usage.
