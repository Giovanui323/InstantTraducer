Il problema segnalato ("non mi ha chiuso il box") è causato da una logica errata nella gestione dello stato di chiusura del pannello di avviso.

Attualmente, il box viene nascosto (dismissed) solo se non ci sono errori "in attesa" (`criticalErrorsCount === 0`). Tuttavia, il box viene mostrato anche quando ci sono errori "in fase di ritentativo" (retrying), situazione in cui `criticalErrorsCount` è 0 ma `totalCriticalCount` è > 0.
Di conseguenza, se l'utente prova a chiudere il box mentre è in corso un ritentativo, il sistema resetta immediatamente lo stato di chiusura (perché vede `criticalErrorsCount === 0`), facendo riapparire il box all'istante.

**Piano di intervento:**

1. Modificare `src/components/ReaderView.tsx`.
2. Aggiornare l'hook `useEffect` che gestisce il reset di `isCriticalRetryDismissed`.
3. Sostituire la dipendenza `criticalErrorsCount` con `totalCriticalCount`.

   * In questo modo, il box rimarrà chiuso (se l'utente lo ha chiuso) finché ci sono errori o ritentativi in corso.

   * Lo stato di chiusura verrà resettato (permettendo al box di riapparire in futuro) solo quando *tutti* i problemi (errori e ritentativi) saranno risolti (`totalCriticalCount === 0`).

Questo risolverà il bug che impedisce la chiusura del box e migliorerà l'esperienza utente ("esce un po' impropriamente").

```typescript
// Codice attuale
useEffect(() => {
  if (criticalErrorsCount === 0 && isCriticalRetryDismissed) {
    setIsCriticalRetryDismissed(false);
  }
}, [criticalErrorsCount, isCriticalRetryDismissed]);

// Codice corretto
useEffect(() => {
  if (totalCriticalCount === 0 && isCriticalRetryDismissed) {
    setIsCriticalRetryDismissed(false);
  }
}, [totalCriticalCount, isCriticalRetryDismissed]);
```
