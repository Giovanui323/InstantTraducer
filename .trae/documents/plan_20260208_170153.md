I will implement a fix to ensure the fingerprint backfilling is persistent. The issue is caused by a race condition where the library refresh (loading data from disk) overwrites the local update (the new fingerprint) before it gets saved, coupled with a delay in the save operation.

### Plan:
1.  **Prioritize Saving**: In `src/App.tsx`, I will change the save priority for the fingerprint backfill from default (Background, 3s delay) to `CRITICAL` (Immediate). This ensures the save request is processed instantly.
2.  **Protect Active Project State**: In `src/hooks/useAppLibrary.ts`, I will modify the `refreshLibrary` function. When updating the list of projects from disk, it will check if there is an active project open. If so, it will preserve the active project's local state (merging it with disk data) instead of blindly overwriting it with potentially stale data from the disk.

This combination ensures that the generated fingerprint is saved immediately and is not lost during the initial library load.
