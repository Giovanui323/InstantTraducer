I have identified the cause of the error log and the timeout issue.

**The Problem:**

1. **Empty Error Log (`{}`)**: The error object is not being serialized correctly when sent from the Renderer process to the Main process. The `Error` object properties (message, stack) are lost during IPC transmission because they are not sanitized/converted to a plain object first.
2. **Timeout Error**: The application forces a timeout after **4 seconds** when saving data during closure. If there are multiple pending saves or disk I/O is slow, this timeout triggers, logging "Errore (o timeout) durante flush in chiusura". The Main process actually waits **10 seconds**, so the Renderer's 4-second limit is unnecessarily aggressive.

**The Plan:**

1. **Fix Logging Serialization**:

   * Modify `src/services/logger.ts` to use the existing `sanitize()` function in `log.error`. This will convert the `Error` object into a plain object (preserving message and stack) before sending it to the Main process, ensuring future errors are logged with full details instead of `{}`.

2. **Extend Flush Timeout**:

   * Modify `src/App.tsx` to increase the flush timeout from **4000ms** to **8000ms**.

   * This gives the application double the time to save pending data while still finishing before the Main process's 10-second hard deadline.

This approach addresses both the root cause of the timeout (by giving it more time) and the observability issue (by fixing the empty log).
