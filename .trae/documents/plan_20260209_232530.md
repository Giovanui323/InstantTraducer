I have identified the cause of the race condition. The `TranslationExecutor` awaits `verifyAndMaybeFixTranslation`, expecting a verification result. However, the UI implementation of this function in `useAppQuality.ts` merely adds the verification task to a background queue and returns immediately (resolving to `undefined`), without waiting for the actual verification to complete. Consequently, `TranslationExecutor` receives `undefined`, interprets it as "no severe error" (OK), and proceeds to log "Quality Check: OK". The actual verification finishes seconds later, logging the "SEVERE" error.

To fix this, I will ensure the verification process is awaited end-to-end.

### Plan:

1.  **Modify `src/hooks/useVerificationQueue.ts`**
    *   Update `enqueueVerification` to return a `Promise` that resolves when the queued task is completed.
    *   Implement a mechanism to store `resolve`/`reject` callbacks for queued items.
    *   Update `pumpQueue` to resolve the corresponding promise with the result returned by `processVerification`.

2.  **Modify `src/hooks/useAppQuality.ts`**
    *   Update `processVerificationInternal` to return the `PageVerification` report (or `undefined`) upon completion.
    *   Update `verifyAndMaybeFixTranslation` to await the result of `enqueueVerification` and return it.

3.  **Verification**
    *   This ensures `TranslationExecutor` will genuinely wait for the quality check to finish.
    *   If the check returns a SEVERE error, `TranslationExecutor` will catch it immediately and behave correctly (skipping save/post-processing), preventing the conflicting logs.
