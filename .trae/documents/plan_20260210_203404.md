I have identified the root causes for all three issues and have a solid plan to fix them.

### 1. Reset Analytics Document
**Problem**: Statistics (success rate, failures) persist between documents because `pdfRenderAnalytics.reset()` is never called.
**Fix**:
- Modify `src/App.tsx` to call `pdfRenderAnalytics.reset()` in two places:
    - Inside `doUpload` (when loading a new file).
    - Inside `handleOpenProject` (when opening an existing project).

### 2. Race Condition Autodetect
**Problem**: The "Auto-Start" translation kicks in immediately after loading, while the AI metadata detection runs in parallel (taking ~12s). If the metadata detection triggers a rename (changing the Project ID), the ongoing translation is left using the old ID, causing pointer misalignments.
**Fix**:
- Modify `src/App.tsx` inside `doUpload`:
    - Explicitly set `setIsPaused(true)` at the start of the upload process.
    - Move the "Unpause" logic (`setIsPaused(false)`) to the *end* of the metadata extraction block.
    - This ensures that while the user can *read* the document immediately, the *automated translation* (prefetch) waits until the Project ID is finalized.

### 3. Persistence PDF Asset vs JSON
**Problem**: The rename operation fails because it expects the JSON file to exist on disk. However, the initial save is currently triggered with `BACKGROUND` priority, which is debounced (delayed) by the library.
**Fix**:
- Modify `src/App.tsx` inside `doUpload`:
    - Change the initial `library.updateLibrary(...)` call to use `'CRITICAL'` priority.
    - This forces the save to happen immediately (skipping the debounce queue), ensuring the JSON file exists on disk *before* the rename operation attempts to access it.

I will apply these changes to `src/App.tsx`.
