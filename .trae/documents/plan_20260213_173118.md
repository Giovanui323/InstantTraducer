I will fix the race condition in the library refresh logic that causes new uploads to be treated as "recovery" sessions.

The issue occurs because `refreshLibrary` can sometimes run (e.g. triggered by metadata updates or UI events) *after* the new project is created in memory but *before* it has been fully saved to the disk. Currently, `refreshLibrary` trusts the disk state implicitly, wiping out the "memory-only" new project. When the next save attempt occurs, the system sees the active ID but can't find the project in the library, triggering the "Recovery" alarm.

I will modify `src/hooks/useAppLibrary.ts` to ensure that if the currently active project exists in memory but is missing from the disk data during a refresh, it is preserved in memory instead of being discarded.
