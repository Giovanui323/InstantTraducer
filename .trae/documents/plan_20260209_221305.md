I will implement a resource cleanup mechanism in the translation executor to prevent "zombie processes" and memory leaks after critical errors (like Global Timeout).

### Problem Analysis
*   **Symptom**: After a Global Timeout (240s), retries may encounter "dirty" state or memory issues.
*   **Root Cause**: The `pdfjs` Page object is created in `executePageTranslation` but is never explicitly cleaned up. While the canvas is cleared, the underlying PDF page resources (textures, text content) remain allocated until Garbage Collection, which may be too late or fail if references persist.
*   **Risk**: Accumulation of unreleased page resources ("zombie processes") leading to RAM exhaustion or file access conflicts during retries.

### Implementation Plan
I will modify `src/services/translation/TranslationExecutor.ts` to:
1.  **Elevate Scope**: Move the `page` variable declaration to the top level of `executePageTranslation` so it's accessible in the `finally` block.
2.  **Explicit Cleanup**: Add a `page.cleanup()` call in the `finally` block. This ensures that regardless of success, error, or timeout, the PDF page resources are immediately released.
3.  **Safe Execution**: Wrap the cleanup in a try-catch to prevent cleanup errors from masking the original error.

### Code Changes
**File**: `src/services/translation/TranslationExecutor.ts`

```typescript
// Before
try {
  // ...
  if (!imageData) {
    const page: any = await pdfDoc.getPage(targetPage);
    // ...
  }
} finally {
  if (canvas) { ... }
}

// After
let page: any = null; // Lifted scope
try {
  // ...
  if (!imageData) {
    page = await pdfDoc.getPage(targetPage);
    // ...
  }
} finally {
  if (canvas) { ... }
  // Explicit PDF page cleanup
  if (page) {
    try { page.cleanup(); } catch (e) { log.warn("Page cleanup failed", e); }
  }
}
```

This change directly addresses the user's concern about "Reset della memoria a breve termine del renderer" and ensures a clean state for retries.
