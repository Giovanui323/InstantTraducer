I have confirmed the issues in the codebase. Here is the plan to fix the file system integrity, race conditions, and silent data loss.

### 1. Fix Race Condition in `rename-translation`
**File:** `electron/projectHandlers.js`
- **Problem:** `rename-translation` attempts to read the existing JSON file immediately. If `save-original-pdf-buffer` or the initial project creation (`save-translation`) is still pending in the `writeSequencer` or OS buffer, `fs.promises.readFile` fails with `ENOENT`.
- **Fix:** 
  - Add a "Pre-check Loop" in `rename-translation` that waits (up to 2 seconds) for the source file to exist before attempting to read it.
  - This ensures that if the renderer triggers a rename immediately after creation, the main process waits for the file to appear.

### 2. Fix Silent Data Loss in Background Jobs
**File:** `src/services/translation/TranslationJobRunner.ts`
- **Problem:** The background runner bypasses the `useAppLibrary` blacklist checks and calls Electron directly. If the save fails (e.g., file locked or write error), it logs the error but **continues execution**, discarding the translated data and moving to the next page.
- **Fix:**
  - Modify `saveWithPriority` to return a distinct success/failure status.
  - In `processPage` (via the `updateLibrary` adapter), check this status.
  - If saving fails, throw a `SaveError` that aborts the current page processing and **Pauses** the background job. This prevents burning tokens when data cannot be saved.

### 3. Fix "Ghost File" Blacklist Persistence
**File:** `src/hooks/useAppLibrary.ts`
- **Problem:** The blacklist mechanism (`blockSave`) protects the renderer state, but if a rename fails (e.g., due to the race condition above), the cleanup might be inconsistent, or the `flushSaves` might try to force-save a blacklisted ID.
- **Fix:**
  - Update `flushSaves` to return a `boolean` (success/failure) instead of `void`.
  - Ensure `flushSaves` respects the `transitioningIdsRef` (don't flush if we are in the middle of a rename) or handles the error explicitly.

### 4. Fix Closure Handshake (False Success)
**File:** `src/App.tsx` & `src/hooks/useAppLibrary.ts`
- **Problem:** `flushSaves` catches errors internally and returns nothing. `App.tsx` assumes completion means success and sends `readyToClose` to the main process, closing the app even if data wasn't written.
- **Fix:**
  - Update `App.tsx` to check the return value of `flushSaves`.
  - If flush fails, show a warning dialog ("Data not saved, force close?") or retry, instead of closing silently.

### Execution Order
1.  **Electron Logic**: Fix `projectHandlers.js` to handle the file existence race.
2.  **Renderer Logic**: Update `TranslationJobRunner.ts` to stop on save failure.
3.  **State Logic**: Update `useAppLibrary.ts` and `App.tsx` to correctly handle flush failures.
