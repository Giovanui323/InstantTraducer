I have identified the cause of the performance degradation and memory leak. It stems from two main sources:
1.  **Unreleased "Previous Page" Objects**: In `TranslationExecutor.ts`, when retrieving the context from the previous page (`prevPage`), the page object is loaded into memory but never released (cleaned up), causing a cumulative leak of one page object per translated page.
2.  **Missing Cleanup in Render Utilities**: The low-level rendering functions in `pdfUtils.ts` do not explicitly call `page.cleanup()` after rendering, relying on garbage collection which may be too slow for batch processing.

I will implement a strict resource management strategy to fix this.

### **Plan: Fix Memory Leaks in Rendering Engine**

#### **1. Fix Context Retrieval Leak in `TranslationExecutor.ts`**
*   **Location**: `src/services/translation/TranslationExecutor.ts` (lines 299-311)
*   **Action**: Wrap the previous page text extraction in a dedicated `try...finally` block.
*   **Change**:
    ```typescript
    // Pseudo-code
    let prevPage = null;
    try {
      prevPage = await pdfDoc.getPage(targetPage - 1);
      // ... get text content ...
    } finally {
      if (prevPage) prevPage.cleanup(); // Force release
    }
    ```

#### **2. Enforce Cleanup in `pdfUtils.ts`**
*   **Location**: `src/utils/pdfUtils.ts`
*   **Action**: Add explicit `page.cleanup()` calls to the `finally` blocks of all rendering functions.
*   **Functions to update**:
    *   `renderDocPageToJpeg`
    *   `renderDocPageToObjectURL`
    *   `renderPageToJpegBase64`
    *   `renderPageOntoCanvas`

This will ensure that every time a PDF page is accessed for any reason (rendering, text extraction, etc.), its internal resources are explicitly freed immediately after use, preventing the "1.5s+" render spikes and subsequent crashes.
