I will fix the "Zombie Save" issue by implementing a robust backend blocking mechanism that prevents save operations for blacklisted/transitioning projects, even if they are already in the IPC queue.

### 1. Backend: Enhance WriteSequencer (`electron/state.js`)
- Add a `blockedIds` Set to `WriteSequencer`.
- Add `block(fileId)` method:
    - Adds ID to `blockedIds`.
    - Clears any pending debounce timeouts for this ID.
    - Removes pending saves from memory.
- Add `unblock(fileId)` method.
- Update `enqueue` task execution to explicitly check `blockedIds` before running the write operation. If blocked, return a specific error or skipped status.

### 2. IPC: Register New Handlers (`electron/projectHandlers.js`)
- Register `block-save` IPC handler calling `writeSequencer.block(fileId)`.
- Register `unblock-save` IPC handler calling `writeSequencer.unblock(fileId)`.

### 3. Bridge: Expose API (`electron/preload.cjs` & `global.d.ts`)
- Expose `blockSave` and `unblockSave` functions in `electronAPI`.
- Update TypeScript definitions in `global.d.ts`.

### 4. Frontend: Integrate with Library Hook (`src/hooks/useAppLibrary.ts`)
- Update `blockSave` to call `window.electronAPI.blockSave(fileId)`.
- Update `unblockSave` to call `window.electronAPI.unblockSave(fileId)`.

This ensures that even if a "Zombie Save" request was sent just before the block, the backend will reject it upon execution, preventing the data corruption/overwrite issue during project switching.
