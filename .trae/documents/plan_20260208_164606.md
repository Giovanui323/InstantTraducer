I have confirmed that the data loss is caused by a flaw in the `TranslationJobRunner.ts` saving logic, which is triggered when the system is under load (evidenced by the timeout errors you found).

### **The Critical Flaw**
The application currently uses a "Load-Merge-Save" strategy on the client side:
1.  **Load**: It tries to read the full project file.
2.  **Merge**: It adds the new translation.
3.  **Save**: It overwrites the file.

When the disk is busy (causing the timeout errors you saw), the **Load** step fails or returns empty data. The application then assumes the project is empty, "merges" the new page, and **overwrites the file, erasing all previous pages (1-11).**

### **The Fix**
I will modify `src/services/translation/TranslationJobRunner.ts` to remove this risky logic. Instead of loading and merging on the client, I will send **only the new translation (delta)** directly to the backend. The backend (Electron) already has a safe, atomic system to merge updates into the existing file without losing data, even if the client has trouble reading.

### **Plan Steps**
1.  **Edit `src/services/translation/TranslationJobRunner.ts`**:
    -   Refactor `saveWithPriority` to remove the `loadTranslation` call.
    -   Change the logic to pass the `data` payload directly to `window.electronAPI.saveTranslation`.
    -   This delegates the merge responsibility to the safe backend process, preventing data loss during read failures.

This directly addresses the "reset" behavior you observed where previously translated pages disappear.
