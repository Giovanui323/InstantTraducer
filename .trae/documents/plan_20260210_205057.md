I will optimize the translation queue to allow **pipelining**: subsequent pages will start **pre-rendering** (loading images, rendering PDF) while the previous page is still being translated by the AI. This removes the "Total Block" and speeds up the process significantly.

### 1. Enable Concurrent Processing (Relax Dependencies)
I will modify `useAppTranslation.ts` to relax the strict `checkDependency` logic.
- **Current Behavior:** If Page `N-1` is processing (`inFlight`), Page `N` waits.
- **New Behavior:** Page `N` will be allowed to start even if Page `N-1` is still processing. This allows Page `N` to perform expensive image rendering operations in parallel with Page `N-1`'s AI request.

### 2. Implement Smart Synchronization
I will modify `executePageTranslation` in `TranslationExecutor.ts` and `useAppTranslation.ts` to handle the synchronization safely:
- **Add `getLatestTranslationMap` service**: Pass a way to access real-time translation updates to the executor.
- **Add "Wait for Context" Step**: Inside `executePageTranslation`, *after* the image rendering phase but *before* the AI request:
  - Check if the previous page's translation is ready.
  - If not ready and the previous page is still running, **wait** (poll) until it finishes.
  - This ensures we don't send empty context to the AI, while still utilizing CPU for rendering.
- **Safety Fallback**: If the previous page fails or stops running, stop waiting and proceed (or fail gracefully).
