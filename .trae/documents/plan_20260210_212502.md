**Analysis**
The current implementation allows a "Pipelining" optimization where a page can start processing (occupying a concurrency slot) even if the previous page is still in progress (`inFlight`). It relies on `TranslationExecutor.ts` to handle the synchronization (waiting) internally.
However, as observed, this causes the page to occupy an active slot while merely waiting, effectively reducing the concurrency available for other pages and potentially leading to deadlocks or inefficient slot usage ("active: 2/2" but stalled).

**Proposed Changes**
1.  **Modify `checkDependency` in `src/hooks/useAppTranslation.ts`**:
    *   Remove the optimization that allows a page to start if the previous page is `inFlight`.
    *   Force the check to return `false` if the previous page is still processing.
    *   This ensures the page remains in the `WAITING` queue and does not consume an `ACTIVE` slot until its dependency (previous page translation) is fully completed and available.
