
# Piano di Intervento: Risoluzione Deadlock Quality Check

Analizzando il codice e i log forniti, il problema risiede nella condivisione del `globalConcurrency` (semaforo globale) tra Traduzione e Verifica, combinata con la mancanza di un timeout *esterno* robusto per la fase di verifica. Se il provider (Gemini Pro) o la logica di preparazione (es. caricamento immagini) si bloccano, lo slot non viene mai rilasciato, bloccando l'intera coda.

Propropongo un intervento su tre fronti: **Disaccoppiamento**, **Timeout Aggressivo** e **Logging Granulare**.

## 1. Disaccoppiamento Concorrenza (`useVerificationQueue.ts`)
Attualmente, `useVerificationQueue` acquisisce uno slot dal `ConcurrencyManager` globale (`globalConcurrency.acquire('Verification')`). Questo significa che se 2 verifiche si bloccano, occupano tutti gli slot disponibili (default 2), impedendo anche alle traduzioni di procedere.

**Modifica:**
- Rimuovere completamente l'uso di `globalConcurrency` dalla coda di verifica.
- La verifica sarà gestita esclusivamente dal limite interno `MAX_CONCURRENT_VERIFICATIONS` (già presente e impostato a 2).
- **Risultato:** Le verifiche gireranno in parallelo alle traduzioni (max 2 traduzioni + max 2 verifiche), impedendo che un blocco nella verifica fermi la traduzione di nuove pagine.

## 2. Timeout Aggressivo e "Safety Net" (`useVerificationQueue.ts`)
Sebbene esista un timeout nella chiamata API (`AI_VERIFICATION_TIMEOUT_MS`), questo non copre:
- Blocchi nel recupero delle immagini (es. `getContextImageBase64`).
- Loop infiniti o errori non gestiti prima della chiamata API.
- Errori di rete che non rispettano il timeout del provider.

**Modifica:**
- Introdurre una costante `SAFETY_TIMEOUT_MS` (es. 120 secondi).
- Avvolgere la chiamata `processVerification` in una `Promise.race` con un timer di sicurezza.
- Se il timer scatta, la promessa viene rigettata forzatamente con un errore `VERIFICATION_TIMEOUT`, garantendo l'esecuzione del blocco `finally`.

## 3. Logging Granulare e Heartbeat (`useVerificationQueue.ts`)
Per monitorare lo stato "buco nero", aggiungeremo log dettagliati direttamente nel ciclo di vita della coda.

**Modifica:**
- **Start Log**: `[VERIF-QUEUE] Avvio verifica pagina X (slot interni attivi: N/M)`.
- **Heartbeat**: Un `setInterval` che ogni 15 secondi logga: `[VERIF-QUEUE] In attesa verifica pagina X da Y secondi...`.
- **Timeout Log**: `[VERIF-QUEUE] TIMEOUT SICUREZZA scattato per pagina X dopo 120s. Forzo rilascio slot.`.
- **Release Log**: `[VERIF-QUEUE] Slot rilasciato per pagina X.`.

## Implementazione Tecnica

Modificherò `src/hooks/useVerificationQueue.ts` per implementare questa logica:

```typescript
// Pseudo-codice della nuova logica in pumpQueue
const SAFETY_TIMEOUT = 120_000;

// Heartbeat logger
const heartbeat = setInterval(() => {
   log.info(`[VERIF-QUEUE] Pagina ${p} in verifica da ${elapsed}s...`);
}, 15000);

try {
    // NESSUN globalConcurrency.acquire() -> Disaccoppiamento
    
    // Race tra processo e timeout
    result = await Promise.race([
        processVerification(p, ctrl.signal, opts),
        new Promise((_, reject) => setTimeout(() => reject(new Error('TIMEOUT')), SAFETY_TIMEOUT))
    ]);
} catch (e) {
    if (e.message === 'TIMEOUT') log.error(...);
} finally {
    clearInterval(heartbeat);
    activePagesRef.current.delete(p); // Rilascio slot locale
    // NESSUN globalConcurrency.release()
    pumpQueue();
}
```

Questa strategia risolve il deadlock alla radice (disaccoppiamento) e fornisce gli strumenti per diagnosticare eventuali problemi futuri (log e timeout).
