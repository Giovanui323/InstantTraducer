I have analyzed the codebase and identified the cause of the inconsistent cooldowns and retry loops.

### **Diagnosis**
1.  **"Flash Immunity" Bug**: The function `translateWithGemini` explicitly skips the cooldown check for the fallback model `GEMINI_TRANSLATION_FLASH_MODEL` (`gemini-2.5-flash`).
    *   Code: `while (model !== GEMINI_TRANSLATION_FLASH_MODEL && isModelInExtendedCooldown(model))`
    *   **Result**: Even if the Flash model is in cooldown (due to quota limits), the system attempts to use it anyway.
2.  **Retry Loop**: When the Flash model fails (Quota 429), it throws an error. The external system (or user) retries. Since higher-tier models (Pro) are in cooldown, the system immediately falls back to Flash again. Because Flash ignores its cooldown status, it attempts the request again, fails again, and creates an infinite loop of 429 errors.
3.  **Parallel Cooldowns**: Models track cooldowns individually (`gemini-3-pro` vs `gemini-3-flash`). However, Quota errors are often project-wide. When the "last resort" model (Flash) fails, it indicates a global project quota exhaustion, but the system doesn't recognize this as a global state.

### **Proposed Solution**
I will implement a **Global Circuit Breaker** for Gemini to synchronize cooldowns and prevent the retry loop.

#### **1. Implement `GlobalGeminiCooldown` in `geminiService.ts`**
*   Add a global state `globalGeminiCooldownUntil`.
*   **Logic**: If the **fallback model** (`gemini-2.5-flash`) fails with a Quota error, it implies the entire project quota is exhausted. We will trigger a **Global Cooldown** for all Gemini models.

#### **2. Update `translateWithGemini` Logic**
*   **Pre-flight Check**: At the very start of the function, check if the **Global Cooldown** is active. If so, throw a specific `GlobalCooldownError` immediately (skipping any API calls).
*   **Remove Immunity**: Modify the fallback loop to respect the cooldown of the Flash model as well. If the Flash model is reached but is in cooldown, the system should stop and throw an error instead of making a doomed request.

#### **3. Unify Retry & Cooldown Policies**
*   Ensure that Quota errors (429) trigger the long cooldown (20 min), while Timeouts trigger the short cooldown (1 min).
*   The Global Cooldown will use the standard `GEMINI_COOLDOWN_MS` (20 min) to ensure the account has time to recover.
