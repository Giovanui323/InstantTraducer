I will fix the race condition where the PDF document is destroyed while asynchronous translation or rendering tasks are still trying to access it.

### Analysis
The error `Page 10 render failed with error: Invalid page request` occurs because the **PDF cleanup process** (triggered by closing a project or switching files) destroys the `pdfDoc` object, but **active translation/rendering tasks** are still running in the background. These tasks try to call `pdfDoc.getPage()` on a destroyed document, causing the crash.

### Proposed Changes

#### 1. Centralize Error Handling in `usePdfDocument.ts`
I will modify `src/hooks/usePdfDocument.ts` to intercept PDF.js destruction errors in `renderDocPageToJpeg` and `renderPageToJpegBase64`.
- **Action**: Wrap `doc.getPage()` and `page.render()` calls.
- **Logic**: If an error matches known destruction patterns (e.g., "Invalid page request", "Worker was destroyed", "Transport destroyed"), I will silence it or rethrow it as a "cancelled" error. This prevents it from being reported as a corruption or system failure.

#### 2. Update `TranslationExecutor.ts` to Ignore Cleanup Errors
I will modify `src/services/translation/TranslationExecutor.ts` to handle these specific errors gracefully.
- **Action**: In the `executePageTranslation` catch block, check for destruction error messages.
- **Logic**: If detected, treat the error exactly like `signal.aborted` (stop processing without updating the UI with an error status). This will suppress the "render failed" log and the red error badge on the UI during project switching.

#### 3. Update `App.tsx` (Safety Check)
I will verify `App.tsx` error handling in `ensureImageLoaded` to ensure it also respects these destruction errors as cancellations, preventing unnecessary retries or error notifications when the document is closing.
