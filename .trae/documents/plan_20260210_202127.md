I will fix the context loss issue by ensuring that the translation queue waits for the previous page to be fully verified before starting the next page. This prevents the next page from capturing a stale "Severe" error state while the previous page is being retried.

### 1. Modify `useTranslationQueue.ts`
- Add a `dependencyTrigger` prop to the hook.
- Add a `useEffect` that triggers `pumpQueue()` whenever `dependencyTrigger` changes.
- This ensures the queue wakes up immediately when a page's verification status updates.

### 2. Modify `useAppTranslation.ts`
- Accept a new prop `verificationTick` (a simple counter).
- Pass this `verificationTick` as the `dependencyTrigger` to `useTranslationQueue`.
- Update the `checkDependency` function to check the verification status of the previous page:
  - If the previous page is `verifying`, return `false` (Wait).
  - If the previous page is `severe` AND has an active auto-retry, return `false` (Wait).
  - Otherwise, proceed.

### 3. Modify `App.tsx`
- Add a state `verificationTick` (integer).
- Update the existing `useEffect` that syncs `quality.verificationMap` to also increment `verificationTick`.
- Pass `verificationTick` to the `useAppTranslation` hook.

### Why this fixes the issue:
The problem occurred because Page 3 started translating immediately after Page 2 finished its *draft* translation, but before Page 2 finished its *quality check*. Page 3 captured a snapshot of the context where Page 2 was still marked as "Severe" (from the previous failure), causing it to ignore the context. By forcing the queue to wait until Page 2 is "Verified" (or stable), Page 3 will always start with the correct, updated context.
